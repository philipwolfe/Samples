The Microsoft .NET Framework SDK Debugger

No matter how skilled a programmer you are, you are bound to make mistakes once in a while. Tracking down problems in your code can be baffling without the appropriate tool support. Fortunately, the compiled nature of ASP.NET means that debugging web applications is no different than debugging any other managed application, and the .NET Framework SDK includes a lightweight visual debugger that is perfectly suited for this task.

This section describes the steps required to debug ASP.NET applications using the visual debugger provided in this SDK. The .NET Framework SDK visual debugger supports manual-attach debugging of processes on a local development computer. The debugger documentation included in this SDK is your best resource for information about specific debugger features.

Enabling Debug Mode for ASP.NET Applications
Because many parts of an ASP.NET application are dynamically compiled at runtime (.htm and .asmx files, for example), you must configure the ASP.NET runtime to compile the application with symbolic information before the application can be debugged. Symbols (.pdb files) tell the debugger how to find the original source files for a binary, as well as how to map breakpoints in code to lines in those source files. To configure an application to compile with symbols, include a DebugMode attribute on the Compilation section of the config.web file at the application's root directory. 
<configuration>
  <compilation debugmode="true"/>
</configuration>
Important: This setting should only be enabled when you are debugging an application, as it will significantly affect the performance of the application.

Debugging ASP.NET Applications
Once DebugMode has been enabled for the application, you should issue a request to the page you wish to debug. This will ensure that the ASP.NET runtime process (xspwp.exe) is created and the application is loaded into memory.

Follow these steps to begin debugging:

  1. Launch the .NET Framework debugger, DbgUrt.exe.
  2. Use the File...Open menu to open the source file for the page you wish to debug.
  3. From the Debug menu, choose "Programs". The dialog in the figure below will appear.
  4. Find the "xspwp.exe" process and double-click it to bring up the "Attach to Process" dialog.
  5. Make sure your application appears in the list of running applications, and select "OK" to attach.
  6. Close the "Programs" dialog.

Important: When attached to the xspwp.exe process all threads in that process are frozen. Under no circumstances should you debug a live production application, since client requests will not execute normally until the debugger is detached.

Setting Breakpoints
To set a breakpoint in you page, click the left-hand margin on a line containing an executable statement or function/method signature. A red dot will appear where the breakpoint is set. Hover over the breakpoint to ensure that it is appropriately mapped to the correct application instance in the xspwp.exe process.

Re-issue the request to the page from your browser. The visual debugger will stop at the breakpoint and gain the current window focus. From here you can step, set variable watches, view locals, stack info, disassembly, etc. You can see the intrinsic objects on the Page, like Request, Response, and Session by using "this" (C#) or "Me" (VB) in the watch window.

Generating Symbols for Pre-compiled Components
To debug pre-compiled components such as business objects or code-behind files, you must compile with symbolic information prior to debugging. Symbols for assemblies are typically found via a path-based search algorithm. The algorithm used by the PDB Library (mspdb69.dll) to find symbolic information is outlined below.
  1. Search same path as assembly (this is the normal location for pdb files).
    For local assemblies, place the symbols (.pdb files) in the application's /bin directory along with the dlls.
  2. Search path as specified in the PE file (the NB10 debug header).
  3. Search NT symbol file locations (environment variables NT_SYMBOL_PATH and NT_ALT_SYMBOL_PATH).
Note: If symbolic information cannot be found, the debugger will prompt for a user-specified location.

Section Summary

  1. The .NET Framework SDK visual debugger supports manual-attach debugging of processes on a local development computer.
  2. DebugMode allows the ASP.NET runtime to dynamically compile with symbolic information. Enable this by setting <compilation debugmode="true"/> in the application root's config.web file. The DebugMode setting should only be enabled when you are debugging an application, as it will degrade application performance.
  3. To debug an application, issue a request to a page, attach the debugger to the xspwp.exe process, set breakpoints and re-issue the page request.
  4. When attached to the xspwp.exe process all threads in that process are frozen. Under no circumstances should you debug a live production application, since client requests will not execute normally until the debugger is detached.
  5. To debug pre-compiled components such as business objects or code-behind files, you must compile with symbolic information prior to debugging.
