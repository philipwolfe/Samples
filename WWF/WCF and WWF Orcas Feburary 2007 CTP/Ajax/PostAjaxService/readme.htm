AJAX Service using HTTP POST

This sample demonstrates how the Windows Communication Foundation may be used to create an AJAX service (a service that may be consumed using simple JavaScript code from a web browser client) that utilizes the HTTP POST verb. This sample builds upon the Simple AJAX Service sample – it is recommended you review it first.

Please note that AJAX support in the Windows Communication Foundation is optimized for use with ASP.NET AJAX through the ScriptManager control. When used in this fashion, there is no need to write custom JavaScript code to call the service as in this sample’s ClientPage.htm file – calling a service takes just one line of JavaScript code. For an example of the Windows Communication Foundation used together with ASP.NET AJAX, please see the ASP.NET AJAX Web Service samples.

Note:

The setup procedure and build instructions for this sample are located at the end of this topic.

 

The service is a regular WCF service with no AJAX-specific code. Since the HttpTransferContractAttribute attribute is not applied, the default HTTP verb (“POST”) is used. POST requests may be somewhat harder to construct than GET ones, but they are not cached and thus should be used for all operations that have side effects and thus caching is not appropriate.

[ServiceContract(Namespace = "http://Microsoft.Ajax.Samples")]

    public interface ICalculator

    {

        [OperationContract]

        double Add(double n1, double n2);

        //Other operations omitted…

    }

 

We create an AJAX endpoint on the service by using the webHttpBinding standard binding configured with the Json messageEncoding and the enableWebScript behavior in the service configuration, just as in the Simple AJAX Service sample.

 

This completes the service implementation. Note that unlike with GET, POST services may not be simply invoked from the browser. For example, navigating to http://localhost/NetFx35Samples/service.svc/ajaxEndpoint/Add?n1=100&n2=200 will result in an error, since in the POST case it is expected that the n1 and n2 parameters will be sent in the message body (in the JSON format) and not in the URL.

 

The client webpage, ClientPage.htm, contains JavaScript code to invoke the service whenever the user moves the mouse over one of the operation words on the page. The code to invoke the service constructs a JSON body with the n1 and n2 parameters, and sets the Content-type header to application/json.

var url = "service.svc/ajaxEndpoint/";

url = url + operation;

 var body = '{"n1":';

 body = body + document.sampleForm.num1.value + ',"n2":';

 body = body + document.sampleForm.num2.value + '}';

 xmlHttp.open("POST",url,true);

 xmlHttp.setRequestHeader("Content-type","application/json");

 xmlHttp.send(body);

 

The response from the service is handled just like in the GET case.

 

To set up, build, and run the sample

1. Make sure you have performed the one-time setup steps in the main Readme under the root Sample dir, including

a.      Ensure you have IIS and ASP.NET installed

b.      Set up a NetFx35Samples directory under c:\inetpub\wwwroot and register it as a “NetFx35Samples” vroot and application in IIS setup

2. Build the solution

3. Navigate to http://localhost/NetFx35Samples/ClientPage.htm (do not simply open ClientPage.htm in the browser from the project directory)

4. Ensure you refresh the page in the browser – otherwise, ClientPage.htm from another AJAX sample may be used resulting in errors. The brute-force approach to do this is to run iisreset and re-open the browser
