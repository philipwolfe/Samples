Conversations

This sample shows how a workflow service can have parallel conversations with multiple clients over the same contract. Using this sample you can understand how to correlate messages that are sent for the same contract and same operation but need to be handled differently as they are sent by different clients. In order to achieve the parallel conversations this sample used the Conversation context functionality provided with workflow services.

This sample also uses the async duplex communication so that the workflow service and the client can asynchronously talk to each other. While doing duplex communication between 2 parties the important thing to keep in mind is that the two parties must exchange some context before this communication can happen both ways. The service that initiates the conversation will receive the context in the reply to its message. This is out of the box functionality of WorkflowServices. In order to support the communication from the other service to the initiating service, the initiating service must send its context information in the first message.

 

Note:

The setup procedure and build instructions for this sample are located at the end of this topic.

The scenario that is implemented by this sample is as follows

A customer has sent a request for an order. In response to this the supplier wants to get quotes from 3 different logistics providers to ship the order. The supplier will send a message to all the 3 suppliers. In response each of the logistics providers will send the quote back by invoking the same operation on the same contract. The supplier workflow because it uses the conversation context functionality will be able to correlate the quotes to the correct receive activity to further process it. Once all the quotes are received the lowest quote is picked and sent to the Customer.

As stated above there are 5 parties communicating in this sample

1.     The customer service

2.     The supplier service

3.     Logistics Provider services

a.     The FedEx service

b.     The UPS service

c.     The USPS service

Customer Service

The customer service sends a message to the supplier with the order details. It also sends a context that the supplier service will use later in order to communicate back. The workflow is very simple and contains an initial send activity which will send the purchase order request message with the purchase order and the context as payloads. A receive activity will then wait for the response from the supplier at a later point in time.

The context that is being sent by the customer service is the context associated with the receive activity. If you look at the ‘before Send’ handler of the send activity you will find code that is used to get the context from the receive activity. The receive activity has a GetContext Method which is called in order to get the context.

 

        private void PrepareOrder(object sender, SendActivityEventArgs e)

        {

            this.order.Amount = 1000;

            this.order.OrderId = 1234;

            this.contextToSend = this.ReceiveOrderDetails.GetContext();

        }

 

 

 

The Supplier Service

The supplier service is the one which is going to have parallel conversations with the 3 logistics providers. In order to do this it uses a combination of a unique context token for each conversation as well the async duplex communication method technique of sending the context in the body as is used in the communication between the customer and supplier services.

Before communicating with the logistics providers the supplier service first receives the message from the client service. In this message the context that the supplier service is going to use to communicate back with the customer service is also received. The supplier service workflow applies this context on the ‘Send’ activity that will be used to send a message to the client service. You can see that code in the code handler of the Code activity inside the first receive activity

 

        private void AcceptOrder(object sender, EventArgs e)

        {

            Console.WriteLine("Order Received...");

            this.supplierAck = "Order Received on " + DateTime.Now;

            this.SendOrderDetails.Context = this.customerContext;

        }

 

 

Once that is received the supplier now must initiate the 3 parallel conversations with the 3 logistics providers. Each conversation branch has a ‘Send’ and a ‘Receive’ activity as shown in the picture above. The Send activities in each branch will send the message to one logistics supplier. The corresponding receive activity in that branch must get back the message from the supplier. In order to do that the Send activity in the message payload must send a unique context in the payload. This payload will then be used by the logistics provider in the message header when the message is sent back with the quote.

By default every receive activity has a context which is the root context.  In order to make the receive activity unique, the receive activity defines a context which is scoped to a parent activity other than the root activity. In this sample if you look at the context token property on each receive activity in the parallel you will find that each of them have a unique name and they are scoped to the parallel activity in which they are contained. Now when the send activity within each of the parallel branches gets a context from the receive activity it will get a unique context that is unique to that receive activity. This unique context is also used to create a queue name for the queue on which the receive activity is listening. When the logistics provider sends the quote back with the context in the header it is used by the workflow service dispatch layer to create the appropriate queue name and enqueue the message, which is then looked at by the receive activity.

 

        private void PrepareUPSRequest(object sender, SendActivityEventArgs e)

        {

            Console.ForegroundColor = ConsoleColor.Green;

            Console.WriteLine("RequestShippingQuote from UPS");

            Console.ResetColor();

            this.contextUPS = this.ReceiveQuoteFromUPS.Context;

        }

 

 

Logistics Provider Services

The logistics provider workflows simply receive the request from the supplier. Along with the quote request is the context from the supplier. That context is applied to the send activity that will send the message back to the supplier.

 

        private void AcceptQuoteRequest(object sender, EventArgs e)

        {

            Console.ForegroundColor = ConsoleColor.White;

            Console.WriteLine("Received ShippingQuote Request");

            this.supplierAck = "Working on quote...";

            Console.ResetColor();

            this.SendShippingQuote.Context = this.supplierContext;

        }

 

 

 

To set up Build the project and run

 

By default this sample does not use persistence. If you want persistence you need to add the <WorkflowRuntime> entry in the App.Config file for each of the solutions. The entry will be under the behaviors section. An example is given below.

 

<behaviors>

      <serviceBehaviors>

        <behavior name="ServiceBehavior"  >

          <serviceMetadata httpGetEnabled="false" />

          <serviceDebug includeExceptionDetailInFaults="true" />

          <workflowRuntime name="WorkflowServiceHostRuntime" validateOnCreate="true" enablePerformanceCounters="true">

            <services>

              <add type="System.Workflow.Runtime.Hosting.SqlWorkflowPersistenceService, System.Workflow.Runtime, Version=3.0.00000.0, Culture=neutral, PublicKeyToken=31bf3856ad364e35"

                   connectionString="Data Source=localhost\sqlexpress;Initial Catalog=NetFx35Samples_ServiceWorkflowStore;Integrated Security=True;Pooling=False"

                   LoadIntervalSeconds="1" UnLoadOnIdle= "true" />

            </services>

          </workflowRuntime>

        </behavior>

      </serviceBehaviors>

    </behaviors>

 

As you can see from the example above that there is a <WorkflowRuntime> section under the behaviors section and a SQLPersistence service is being added. You can also see from the connection string that we are using a database called NetFx35Samples_ServiceWorkflowStore. You can use the same database for all the solutions or create separate databases for each. In that case you need to ensure that you install the persistence databases properly. Please consult the WF documentation for details on how to install the databases.

 

The script provided with these samples ‘CreateStored.cmd’ will create the NetFx35Samples_ServiceWorkflowStore database.

 

Once all the workflow services are up and running press enter in the client workflow and see all the workflows communicating with each other.

In order to run on the console window for the Supplier press enter to send the purchased order. As shown in the picture above. The purchase order is sent to the supplier who in turn will send logistics quote requests to all the logistics suppliers. Once the supplier receives quotes you can see that the lowest quote is sent back to the customer. You can also verify the context tokens in each case. This will help you to understand the unique contexts that are sent to each logistics provider.
