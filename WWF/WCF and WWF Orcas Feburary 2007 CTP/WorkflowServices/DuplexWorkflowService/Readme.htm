Duplex Workflow Service

This sample shows how async duplex communication can be done between 2 communicating services. This sample also shows how localhost to workflow communication can also be done using messages. While doing duplex communication between 2 parties the important thing to keep in mind is that the two parties must exchange some context before this communication can happen both ways. The service that initiates the conversation will receive the context in the reply to its message. This is out of the box functionality of WorkflowServices. In order to support the communication from the other service to the initiating service, the initiating service must send its context information in the first message.

Communication with the local host and the service is also done using messages. The local host exposes a well known end point which can be used by the service to call back. The sample has a LocalWorkflowServiceHost type which has the functionality to create the local listener as well as the workflow service host for the service hosted.

 

Note:

The setup procedure and build instructions for this sample are located at the end of this topic.

This sample has 4 communicating parties involved

1.     Client Host

2.     Client Workflow (exposed as a service)

3.     Service Host

4.     Service Workflow (exposed as a service)

The client host and the Client workflow both implement the IReverseContract. The Service Workflow implements the IForwardContract. The details of each component involved is as follows

Client Host

The client host communicates with the Client workflow by calling operations on the IReverseContract. When the Client Host first calls the BeginWork operation that Client workflow is created. As a reply to the BeginWork operation the client workflow sends back a message. The Context for the client workflow is carried on that message. The channel that the host creates to communicate with the client workflow now has the context for further communication. In the sample you can see that the client host uses the same channel to send multiple work items to the client workflow

In order for the workflow to communicate with the client host, the client host has a well known endpoint that it listens on. In this sample that well known endpoint is called ‘HostEndPoint’. The LocalWorkflowServiceHost is used to create that well know end point for the client host as well as create the workflow service host for the client workflow.

 

            LocalWorkflowServiceHost localHost = new LocalWorkflowServiceHost(typeof(ClientWorkflow),new ClientHost());

            localHost.WorkflowRuntime.WorkflowTerminated += delegate(object sender, WorkflowTerminatedEventArgs e) { Console.WriteLine("WorkflowTerminated: " + e.Exception.Message); };

            localHost.WorkflowRuntime.WorkflowCompleted += delegate(object sender, WorkflowCompletedEventArgs e) { Console.WriteLine("WorkflowCompleted: " + e.WorkflowInstance.InstanceId.ToString()); };

            localHost.Open();

 

 

Client Workflow

As stated earlier the client workflow implements the IReverseContract. This is used by the service workflow in the diagram above to communicate back with the client workflow. The client workflow first calls the ‘BeginWorkflow’ operation. This operation caused the service workflow to be created. In return the context for the service workflow is also received back in the reply. This context is used in subsequent communications from the client workflow to the service workflow. As part of the BeginWorflow operation the client workflow is passing its own EndPoint reference which contains the end point address as well as the context headers to the service workflow. The service workflow will use this in order to communicate asynchronously with the Client workflow.

Look closely at the ‘WaitForBeginWork’ event handler in the initial state. You will find that there is a code activity called ‘DoSetReturnAddress’. This activity sets the end point reference which is sent over to the service workflow. Take a

 

        private void SetReturnAddress(object sender, EventArgs e)

        {

            EndpointAddress epr = ContextManager.CreateEndpointAddress(ReturnUri, this.ReceiveWorkItemComplete);

            ReturnAddress = EndpointAddress10.FromEndpointAddress(epr);

            DebugOutput("[ClientWorkflow:SetReturnAddress] " + epr.Headers[0].GetValue<string>());

        }

 

 

Take a look at the Context Manager helper class which has all the functions required to manipulate the context.

 

 

Service Host

The service host implemented here is a simple workflow service host. It simply opens up a listener and listens to requests from the client. The first request from the client will cause an instance of the service workflow to be created. All subsequent requests will be routed to the same workflow instance as those requests will carry the context on the message header.

 

Service Workflow

The service workflow implements the IForwardContract. The Service Workflow in the BeginWorkflow operation receives the end point reference for the client workflow. The service workflow applies that end point reference to the Send activity that is going to send a message asynchronously to the Client workflow. Take a close look at the ‘WaitForBeginWorkflow’ event handler. In it you will find the ‘BeginWorkflow’ receive activity. Inside the receive activity there is a code activity called ‘ApplyReturnAddress’  This activity will apply the return address to the send activity that will send a message to the client workflow.

 

        private void ApplyReturnAddress(object sender, EventArgs e)

        {   // apply ReturnAddress to ReverseEndpoint

            EndpointAddress epr = ReturnAddress.ToEndpointAddress();

            ContextManager.ApplyEndpointAddress(this.SendWorkItemComplete, epr);

            DebugOutput("[ServiceWorkflow:ApplyReturnAddress] " + epr.Headers[0].GetValue<string>());

        }

 

 

LocalWorkflowServiceHost

This class is used in order to create the necessary infrastructure for host to workflow communication. The local workflow service does 2 main things

1.     Creates a local listener for the host to listen to messages from the workflow.

2.     Instantiates the WorkflowServiceHost to create a listener for the workflow

It also provides certain helper functions for the host to maintain the context that is used to communicate with the workflow. In case the host recycles this context will be used to communicate with the workflow instance.

 

ContextManager

Context Manager provides all the helper functions that are required to manipulate the context. It provides functions to extract the context from the channel, apply context to a channel, Apply and end point address to the send activity etc.

 

To set up Build the project and run

In order to install the persistence providers run the command createstored.cmd. This command file is located in the SetUp directory.

If you do not want to use persistence providers then comment out the <WorkflowRuntime> section from the app.config files.

In this sample there are two persistence databases that are used. One used by the Service workflow called NetFx35Samples_ServiceWorkflowStore and one used by the client called NetFx35Samples_ClientWorkflowStore

 

Once the client host and the client service are running press Y in the console window to process workitems. You can send as many workitems for processing. Try shutting down the client and restarting it. You will find that the client will resume from where it left off.

 

Test the Durable nature of the services

 

In order to test the durable nature of the service you can shut both the client and the service applications. Now restart the client application first and then the service application. This is merely for sequencing since as soon as the service workflow is up and running it will try to send messages back to the client and if the client is not around you will get an exception.

 

On the client side the context is stored in a file called ‘Client.ctx’. The file is stored in the \bin directory of your sample. When you re-open the client it looks if the file is present. If it is present it will use the context stored and apply it to the channel that is being created. In case the workflow service has finished and you open the client with the ‘client.ctx’ file still in your \bin directory it will try to apply the context to the channel. You will get an error in this case since the workflow instance to which you want to communicate with is not there. Simply delete the file and try again.
