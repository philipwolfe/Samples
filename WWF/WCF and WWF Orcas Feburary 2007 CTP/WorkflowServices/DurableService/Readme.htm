Durable Workflow Service

This sample demonstrates how to create a durable service. The sample shows the implementation of a simple calculator as a durable service.  Since the service is durable after every call the data state of the service is saved, thus if the service recycles then its state can be restored to the last update point. The sample requires a persistence service to run. This note shows how the persistence service is specified.

 

Note:

The setup procedure and build instructions for this sample are located at the end of this topic.

The ICalculator Service contract

 

[ServiceContract(Namespace = "http://Microsoft.WorkflowServices.Samples")]

    public interface ICalculator

    {

        [OperationContract()]

        int PowerOn();

        [OperationContract()]

        int Add(int value);

        [OperationContract()]

        int Subtract(int value);

        [OperationContract()]

        int Multiply(int value);

        [OperationContract()]

        int Divide(int value);

        [OperationContract()]

        void PowerOff();

    } 

 

Service Implementation

   [Serializable]

    [DurableServiceBehavior]

    public class DurableCalculator : ICalculator

    {

        int currentValue = default(int);

 

        [DurableOperationBehavior(CanCreateInstance=true)]

        public int PowerOn()

        {

            return currentValue;

        }

 

        [DurableOperationBehavior()]

        public int Add(int value)

        {

            return (currentValue += value);

        }

 

        [DurableOperationBehavior()]

        public int Subtract(int value)

        {

            return (currentValue -= value);

        }

 

        [DurableOperationBehavior()]

        public int Multiply(int value)

        {

            return (currentValue *= value);

        }

 

        [DurableOperationBehavior()]

        public int Divide(int value)

        {

            return (currentValue /= value);

        }

 

        [DurableOperationBehavior(CompletesInstance=true)]

        public void PowerOff()

        {

        }

 

    }

If you look at the code for the service implementation there are several things to note

1.    The DurableServiceBehavior attribute is used to specify that this particular service implementation is a durable service. You can look at it to see which other attributes can be specified on the durable service. There are two ther attributes that can be specified

a.     The persistence provider can be specified in the attribute. In this example the persistence provider is specified in the config file.

b.    SaveInstanceStateInTransaction specifies that the state of the instance is saved in the database as part of the transaction of the instance

2.    The durableoperation behavior attribute specifies that the instance state should be saved after the operation is complete.

a.     The CanCreateInstance property specifies that the instance will be created when that operation is invoked

b.    The CompletesInstance property specifies that the instance will be completed after that operation completes. The state is also deleted from the database after this.

 

Take a look at the web.config file to see the bindings for the service you will find the there are two different endpoints with two different bindings that are exposed

1.    A Basic HTTP binding with Cookies enabled is provided. This can be used by client that do not use the NetFx provided HTTP context binding

2.    The wsHTTPContextBinding is the out of box context binding that help to maintain the context that is used to route the requests to the particular workflow instance.

3.    Under ServiceBehaviors you will see that there is a persistenceProvider entry where the persistence provider is specified.

 

To set up the service in IIS

  1. Make you previously ran the setupvroot.bat script located in the Setup directory.

     2. Enable "Windows Authentication" on the vroot "NetFx35Samples". You need to do this in IIS Console. Right click on the Vroot NetFx35Samples and go to the Directory Security tab. Click on the Edit button under Authentication and Access Control and then select Integrated Windows Authentication

           

     3. Project will build and copy to vroot NetFx35Samples.

   

To allow access to durable store

 

    1. Make sure you ran the createstores.cmd script located in the Setup directory. This sample uses the NetFx35Samples_DurableServiceStore database

    2. Make you have ASPNET User account a member of SQLServer*SQLExpress User Group.

 

To ensure that the service has been property installed you can look point your browser at the address http://localhost/netfx35samples/service.svc . You should be able to see the help page for the service. To look at the wsdl type in  http://localhost/netfx35samples/service.svc?wsdl

 

 

To access the service from a client

In order to run this sample you need to use the CalculatorClient sample. It is a calculator UI created using WPF and acts as a client for the service. You can use different end points each corresponding to the bindings provided by the service. In order to change the bindings click on the EndPoint menu item and select the appropriate binding. You have a choice of two bindings a basic http binding and a wsHttpContextBinding

 

Test the Durable nature of the service

 

While the calculator client is working you can close the calculator client and open it again. You will find that the calculator client can communicate back to the same service instance. It displays the instance id at the bottom. The calculator client uses a text file called client.ctx to store the context in a durable place the first time a call is made. The file is stored in the \bin directory of your sample. When you re-open the client it looks if the file is present. If it is present it will use the context stored and apply it to the channel that is being created. In case the workflow service has finished and you open the client with the client.ctx file still in your \bin directory it will try to apply the context to the channel. You will get an error in this case since the workflow instance to which you want to communicate with is not there. Simply delete the file and try again.

 

You can also recycle the workflow service. To do this simply recycle IIS. Since you are using a persistence store after every operation the state of the service is stored, so when you try to communicate with the service from the client after IIS has started again, the workflow infrastructure will get the workflow instance from the persistence store and you can communicate with the same instance again. There is a know issue here, When you invoke an operation for the first time after recycling IIS you will get a MessageSecurityException. Simply invoke another operation and it will succeed. The exception is caused by an expired security token on the channel.
