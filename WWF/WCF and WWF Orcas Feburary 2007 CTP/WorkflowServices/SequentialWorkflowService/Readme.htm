Sequential Workflow Service

This sample demonstrates how to create a workflow service using a sequential workflow. This sample shows how a service contract can be created in place. This method of service authoring is referred to as the ‘Workflow First’ mode of service creation. The sample creates an ICalculator contract and is implemented in the workflow as the workflow is being built.

Since the calculator is a workflow several other benefits can be seen here.

1.     A long running process where the state is saved automatically. You can restart eh workflow service and the client can communicate with the same instance on restart and the state of the workflow would be maintained

2.     The application level protocol in the contract can be enforced. i.e. if an operation is called on this service where the workflow is not expecting that operation to be called the workflow will raise an exception.

The setup procedure and build instructions for this sample are located at the end of this topic.

In this workflow the powerOn Receive activity has a property called ‘CanCreateInstance’ which is marked as true. What this means is that an instance of this service will be created when the PowerOn operation is called by the client. The client receives back the context as part of the reply for this operation. From that point onwards any of the other operations in the workflow can be called. On each operation the client calls the ‘context’ will be carried on the channel and hence the operation can be routed to the correct instance. When the powerOff operation is called a Boolean value is set which will cause the workflow to exit the while loop.

If you look at the program.cs file in the service project you will find the workflowServiceHost invocation pattern. The code there is

WorkflowServiceHost workflowHost = new WorkflowServiceHost(typeof(Microsoft.WorkflowServices.Samples.SequentialCalculatorService));

workflowHost.Open();

 

Here the workflowservice host takes input the workflow type as input and the opens it to listen to any messages.

This sample also contains a client application which is used to invoke operations on the calculator service. This client application is a simple sequential workflow and makes a series of operation invocations. The client workflow uses Send activities to invoke operations on the workflow. Note that the client project contains references to the Icalculator Service. With this reference Send activity can now show the interface from which appropriate operations can be selected for invocation.

 

 

To set up Build the project and run

In order to install the persistence providers run the command createstored.cmd. This command file is located in the SetUp directory. This sample uses the NetFx35Samples_ServiceWorkflowStore database. There is no persistence store used by the client workflow in this sample.

If you do not want to use persistence providers then comment out the <WorkflowRuntime> section from the app.config files.

Once you start running you will find that the client workflow simply send a set of calculator operations and then completes the workflow. You can press enter to send another set of calculator operations.

 

Test the long running nature of the service

You can test the long running nature of this service as follows. Put a break point in the client application. When you hit the breakpoint in VS close the service application and then restart it. Now on the client release the break point. You will find that the client workflow talks to the same workflow instance on the service side. You may want to break the sample into 2 solution to test the recycling of the service easily.
